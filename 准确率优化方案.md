# 准确率问题分析与优化方案

## 问题案例汇总

### 案例1: 500.com 赛事分析页
```
URL: https://live.m.500.com/detail/football/1182860/analysis/zj
实际发布时间: 2025/8/11 3:00:00
当前提取结果: 2026-05-12 21:02:47 (当前时间！)
问题: 浏览器渲染时提取到页面加载时间，不是真实发布时间
```

### 案例2: 微博详情页
```
URL: https://m.weibo.cn/detail/5297296482500834
实际发布时间: 2026/5/11 7:56:00
当前提取结果: 2026-05-05 00:00:00
问题: 正则匹配到文本中的其他日期，不是发布时间
```

### 案例3: 微信百科页
```
URL: https://baike.weixin.qq.com/v10936839.htm
实际情况: 百科页面，无发布时间概念
当前提取结果: 1973-02-06 00:00:00
问题: 提取到人物生日（"1973年2月6日"），误判为发布时间
```

### 案例4: 球迷屋新闻
```
URL: https://m.qiumiwu.com/news/1935754101879
实际发布时间: 2026/5/11 1:59:00
当前提取结果: 2026-05-10 23:30:00
问题: 时间偏差1.5小时，可能是时区或修改时间
```

---

## 优化方案

### 1. **过古时间过滤**（类似未来时间）
```python
问题: 案例3提取到1973年（53年前）
方案: 如果时间 < 当前时间-20年 → 判定为无效
原因: 正常发布时间不会超过20年前
```

### 2. **"刚刚"时间过滤**
```python
问题: 案例1提取到当前时间（页面加载时间）
方案: 如果时间与当前时间相差 < 5分钟 → 判定为无效
原因: 很可能是页面加载时间，不是发布时间
```

### 3. **优先级规则调整**
当前问题：正则匹配优先级太高，容易误匹配
```python
当前优先级:
1. meta标签
2. JSON-LD
3. time标签
4. data属性
5. class/id
6. 正则提取 ← 容易误匹配

优化后优先级:
1. meta标签 (最可靠)
2. JSON-LD (结构化数据)
3. time标签 (语义明确)
4. data属性 (专门存时间)
5. class/id (需要准确匹配)
6. 正则提取 ← 降到最后，且增加过滤规则
```

### 4. **百科/人物页识别**
```python
问题: 案例3是百科页，不应该有发布时间
方案: 页面特征评分增加规则
- URL含 /baike/ /wiki/ /person/ → -20分
- 标题含 "百科" "词条" → -10分
- 页面含 "出生" "生日" 关键词 → -10分
```

### 5. **正则提取增强过滤**
```python
当前问题: 正则太宽松，匹配到任何年月日格式
优化方案:
1. 排除明显的人物生日上下文:
   - "出生", "生于", "诞生", "生日"
   - "born", "birthday", "birth date"
   
2. 排除页面元素中的固定文本:
   - 版权声明: "© 2024"
   - 页脚信息: "2024年3月"
   
3. 优先匹配有明确前缀的时间:
   - "发布时间:", "发表于:", "更新时间:"
   - "publishTime:", "createTime:", "postDate:"
```

### 6. **微博等社交平台专项优化**
```python
问题: 微博页面结构复杂，时间藏在JS数据中
方案:
- 优先提取JSON数据中的时间戳
- 匹配微博特有的时间格式
- 使用浏览器渲染提取动态加载的时间
```

- 阈值从5分钟调整为60秒
- 只过滤"此时此刻"的时间，不误杀新鲜内容

**代码**：
```python
def is_likely_access_time(self, dt_string):
    """
    判断是否可能是页面访问时间（而非真实发布时间）
    只过滤与当前时间相差<60秒的时间
    """
    parsed = datetime.strptime(dt_string, '%Y-%m-%d %H:%M:%S')
    now = datetime.now()
    diff = abs((now - parsed).total_seconds())
    return diff < 60  # 只过滤"此时此刻"
```

## 核心改进总结

### 合理性评估 ✅

| 优化项 | 是否trick？ | 合理性 |
|--------|------------|--------|
| 过古时间过滤(20年) | ❌ 不是 | 发布时间不会是20年前 |
| 未来时间过滤(1天) | ❌ 不是 | 发布时间不可能在未来 |
| 页面访问时间检测(60秒) | ❌ 不是 | 发布时间不会是"此刻" |
| Script变量提取 | ❌ 不是 | SPA应用的标准提取方式 |
| 百科/人物页识别 | ❌ 不是 | 基于页面结构特征判断 |

**结论**：所有优化均为合理方案，不是trick，具有普适性。

### 案例4：球迷屋新闻

**问题**：时间偏差1.5小时（预期1:59，实际23:30）

**可能原因**：
1. **提取到错误字段**：可能是更新时间、评论时间、作者发布时间等
2. **时区问题**：23:30 + 2.5小时 = 1:59（但时区差不应该这么奇怪）
3. **多个时间字段**：页面有多个时间，我们提取错了

**建议**：
- 人工查看HTML源码，确认真实发布时间的准确位置
- 如果页面确实没有准确发布时间，调整页面特征评分，降低为"提取失败"

**是否需要继续优化？**
